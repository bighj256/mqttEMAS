# MQTT环境监测系统 (mqttEMAS2) - 开发与扩展建议 TODO 表格

本项目目前已经打下了非常好的基础，具备了 MQTT 连接管理、多格式数据解析（JSON/CSV/KV）、日志记录以及多窗口查看历史和实时数据的功能。代码结构也很清晰。为了进一步提升系统的实用性和完善度，我为您整理了以下可以继续扩展和优化的功能点（TODO List）：

## 🌟 核心功能扩展 (High Priority)

- [x] **SQL Server 数据持久化存储 (SQL Server Database Storage)**
  - *当前状态*: 历史数据存储在 `Widget::globalHistory` (最大500条的内存 List) 中，软件关闭后数据丢失。
  - *扩展建议*: 由于项目后续可能接收海量传感器数据，建议使用 **SQL Server** 构建高可用的中心化数据库。通过配置 ODBC 并调用 `QSqlDatabase::addDatabase("QODBC")`，将不同 `deviceId` 的数据分表或归档存入系统，为大数据挖掘与分析打下基础。考虑到报表导出需求，该库同样可以扩展“导出为 CSV/Excel”支持。
- [x] **数据可视化图表 (Data Charts)**
  - *当前状态*: 数据仅以文本或列表的形式展示。
  - *扩展建议*: 使用 `QtCharts` 模块或第三方的 `QCustomPlot` 库，在界面上绘制温度、湿度的**实时动态折线图**，让环境趋势一目了然。
- [x] **系统配置保存 (Settings Persistence)**
  - *当前状态*: 每次启动需重新输入参数（目前写死了默认 IP），“系统设置”按钮提示正在开发中。
  - *扩展建议*: 使用 `QSettings` 将上次成功连接的服务器地址、端口、主题等信息保存到本地注册表或 `.ini` 文件中，下次启动时自动读取加载。

## 🛡️ 可行性与业务逻辑提升 (Medium Priority)

- [ ] **阈值报警与通知 (Alarms & Notifications)**
  - *扩展建议*: 允许用户在设置窗口中针对每个传感器设定温度/湿度的上下限安全阈值（如：温度持续 > 40°C 则报警）。
  -触发报警时可以通过弹窗、发出提示音 ( `QSoundEffect` )，或在系统托盘 (`QSystemTrayIcon`) 弹出消息通知。
- [ ] **多设备/多节点分类管理 (Multi-Device Management)**
  - *当前状态*: `DataParser` 已经能够解析出 `deviceId`，但主界面展示可能没有按设备严格区分。
  - *扩展建议*: 增加一个“设备列表”侧边栏或表格视图（QTreeView / QTableView），当不同 `deviceId` 接入时自动注册设备节点，点击某个设备即可查看该设备独立的温湿度数据和历史记录。
- [ ] **反向控制与下发指令 (Command Publishing)**
  - *扩展建议*: 目前仅实现了订阅(Subscribe)接收数据。可以增加一套**下发面板**功能，通过 MQTT Publish 功能向特定的 Topic 发送控制指令（例如控制继电器开关、重启前端传感器、改变采样频率等）。

## 🔧 网络与安全性强化 (Advanced)

- [ ] **完善认证机制 (Authentication & TLS)**
  - *扩展建议*: 增加用户名(Username)和密码(Password)输入框；并考虑引入 SSL/TLS 加密连接能力（使用证书和 `mqttClient->connectToHostEncrypted()`），适应公网环境中更高级的安全需求。
- [x] **断线自动重连机制 (Auto-Reconnect)**
  - *扩展建议*: 虽然有错误处理，但可以增加一个定时器 (`QTimer`)，在网络波动导致意外断开后，尝试每隔 5 秒自动重连，提高系统的无人值守稳定性。

## 💻 代码层面的优化建议 (Code Optimization)

- [ ] **UI 与业务逻辑进一步解耦**
  - 目前 `Widget` 类承担了 MQTT 连接控制、界面更新、日志管理等多个职责。后期如果项目变大，可以考虑将 MQTT 通信封装成一个独立的 `MqttManager` 单例类或工作线程，主窗口只负责绑定信号槽来刷新 UI。
- [ ] **历史数据模型优化**
  - 使用 `QStringList` 保存 `19:23:45 | 温度: ...` 这种格式虽然简单，但对于后续过滤和搜索不够友好。建议将其改造为一个包含结构化 `SensorData` 的实体列表类或封装在 `QAbstractTableModel` 中，以支持 MVC 模式下更灵活的 UI 排序和过滤。

## 🔌 硬件端优化建议 (Hardware / ESP-01S + 8051)

- [ ] **WiFi 与 MQTT 动态配置 (SmartConfig / Web 配网)**
  - *当前状态*: 在 `main.c` 中写死了 WiFi (OnePlus Ace 5) SSID/密码和 MQTT 服务器 IP。若实际部署环境改变，必须重新修改代码并烧录 8051 芯片。
  - *扩展建议*: 建议利用 ESP-01S 的能力，如果连接失败，或者通过单片机按键触发，进入 SmartConfig 或 SoftAP 模式，实现**手机端动态配网**功能，并将这些参数保存在 EEPROM 中。
- [ ] **双向通信与状态机处理 (Bi-directional MQTT)**
  - *当前状态*: 8051 仅使用 `SendString` 盲发 AT 指令，没有开启串口接收中断，因此无法接收服务器下发的指令。目前系统只实现了“上报”而没有“下发”。
  - *扩展建议*: 在 `ESP-01S.c` 中加入 UART RX (串口接收) 中断服务。当收到 ESP8266 传回的 `+MQTTSUBRECV` 信息时解析控制指令（如：控制某路继电器通断）。同时解析 `OK`/`ERROR` 也能帮助单片机判断是否断网并执行自动重连机制。
- [ ] **非阻塞式延时与任务调度 (Non-blocking Architecture)**
  - *当前状态*: `while(1)` 循环中大量使用 `Delay_ms(3000)` 进行强制等待，锁死了 CPU 资源，使得系统无法及时响应突发事件（如串口接收或外部急停按键）。
  - *扩展建议*: 基于 8051 的定时器中断（如 Timer0）生成全局时间戳 (Tick)，将主循环改造为**状态机并发模式**或引入简单的轮询调度框架，从而大幅提升硬件底层的实时并发性能和代码健壮性。
